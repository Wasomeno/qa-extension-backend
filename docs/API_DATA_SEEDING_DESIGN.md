# Fast-Track Setup via API (Data Seeding)

## 1. Core Objective
The goal is to implement a **Fast-Track Setup** flow for automated test generation. 
Instead of generating slow and brittle UI navigation steps to fulfill test preconditions (e.g., logging in, creating prerequisite data records), the AI will generate direct HTTP API requests. 
The test runner will execute these API calls to seed the necessary data and state, and only switch to UI actions (`navigate`, `click`, `type`) for the actual test execution steps.

## 2. API Source of Truth (Frontend-Driven Approach)
We are using **Option A (Frontend Codebase Parsing)** to provide API knowledge to the LLM. 
Instead of relying on a static backend Swagger/OpenAPI JSON, we will feed the relevant frontend API client files to Gemini.

* **Files Provided to LLM:** 
  * API Call Definitions (e.g., `src/api/auth/api.ts`)
  * TypeScript Types/Interfaces (e.g., `src/api/auth/type.ts`)
  * Route Constants (e.g., `src/commons/constants/routes.ts`)
* **LLM Responsibility:** Gemini will read these files, connect the TypeScript interfaces to the constants, and autonomously construct the correct endpoints and JSON payloads for the `api_request` steps.

## 3. Data Model & Schema Updates
To support API requests alongside standard UI actions, the core data models and Gemini output schemas must be extended.

### Backend Struct Changes (`internal/models/manual_recording.go`)
The `RecordingStep` struct is expanded to include API-specific fields:
* `Action`: Added `"api_request"` to the enum.
* `ApiMethod`: `GET`, `POST`, `PUT`, `DELETE`, `PATCH`.
* `ApiEndpoint`: The resolved URL path (e.g., `/auth/sign-in`).
* `ApiPayload`: Stringified JSON body containing the request data.
* `ApiHeaders`: Stringified JSON of required headers.

### Gemini Schema Changes (`services/test_generator.go`)
The strict Vertex AI JSON generation schema is updated to include the new fields and allow `"api_request"` as a valid action type so the LLM can generate them directly in the steps array.

## 4. Prompt Engineering Strategy
The AI prompt (`buildCatalogAwarePrompt`) will be updated with explicit behavioral instructions:
* **Precondition Handling:** When evaluating the `PreCondition` field from the provided XLSX test scenarios, the AI must explicitly avoid using UI interactions.
## 5. Token Passing & State Management
Because projects use different client-side state management (e.g., Vite/React with `js-cookie` vs. Next.js with `NextAuth`), the test generator (LLM) will remain framework-agnostic. 

* **Variable Placeholder:** The LLM prompt will instruct Gemini to use a standardized placeholder `{{AUTH_TOKEN}}` whenever an API request requires a token (e.g., `{"Authorization": "Bearer {{AUTH_TOKEN}}"}`).
* **Runner Auto-Extraction:** The test runner (Playwright/Cypress) will dynamically inspect the HTTP response of the initial login `api_request`. It will extract the token from the response body (e.g., `response.data.token`) and substitute `{{AUTH_TOKEN}}` in all subsequent `api_request` steps.
## 6. Target-Driven Context Fetching
To ensure the LLM has the necessary API and constant files without exceeding the 30-file / 80k-token codebase context limits, the backend file fetcher (`codebase_fetcher.go`) will use **Targeted Dependency Fetching**.

Instead of purely prioritizing UI folders (`pages/`, `components/`), the fetcher will extract a target keyword (e.g., `auth`, `cash-ins`) from the test scenario. It will sort the GitLab file tree to prioritize:
1. Universal Routing/Endpoint Constants.
2. Exact-match API files (e.g., `src/api/{keyword}`).
3. Exact-match UI files (e.g., `src/pages/{keyword}`).

## 7. Dynamic API Chaining & Payload Generation
Because the testing methodology relies on "Full Flow" creation (creating prerequisite data from scratch), the test generation will heavily utilize API request chaining rather than hardcoded mock IDs.

* **Response Variable Extraction:** The LLM is instructed to use a `{{STEP_X_RESPONSE.path}}` syntax. The test runner will parse API responses in memory and inject them into subsequent steps (e.g., extracting a newly created `supplier_id` to use in an `invoice` creation step).
* **UI-Driven Lookup Emulation:** By analyzing the frontend React code (`src/pages/*.tsx`), the LLM will identify when a form requires reference data (e.g., dropdowns populated by `GET /taxes`). It will generate the corresponding `GET` requests to retrieve valid IDs from the staging environment before generating the `POST` payload.
## 8. Test Runner Handoff (Execution Contract)
The JSON generated by the AI (via the Go backend) must be executed by a downstream Playwright/Cypress runner. The Go backend enforces a strict contract for this handoff.

1. **Framework Metadata:** The Go `codebase_fetcher` will detect the frontend framework (`vite.config.ts` vs. `next.config.js`) and inject a `"framework"` field into the top-level `GeneratedAutomation` output. The runner uses this to determine the authentication injection strategy (e.g., `js-cookie` vs NextAuth `storageState.json`).
2. **Dynamic Base URL:** The API Base URL is not hardcoded by the LLM. The Test Generation UI will provide an input field for the user to specify the target API environment. The Go backend passes this value to the runner, which prepends it to the LLM-generated `apiEndpoint`.
3. **Execution Pipeline:** The runner is responsible for:
   * Initializing a runtime memory store (`stepResponses = {}`).
   * Hydrating `{{STEP_X_RESPONSE.path}}` variables before firing HTTP requests.
   * Executing the `api_request` and storing the JSON response in memory.
   * Pausing execution before the first UI `navigate` action to inject the authentication state into the browser context based on the `"framework"` metadata.
