# API Data Seeding: Implementation Checkpoints

This document tracks the resolution of the four core implementation challenges required to make the API Data Seeding flow work.

## [x] 1. Token Passing & State Management
*Status: Resolved*

**Solution:** Use an LLM-agnostic placeholder system (`{{AUTH_TOKEN}}`). The test runner extracts the token from the login API response and substitutes it in subsequent API calls. The runner handles injecting state into the browser (via cookies for Vite or `storageState.json` for Next.js).

## [x] 2. File Fetching Limits
*Status: Resolved*

**Solution:** Implement **Targeted Dependency Fetch (Option B)** in the Go backend (`codebase_fetcher.go`). 
Instead of blindly sorting files by type (e.g., all `pages` first), the fetcher will accept a `targetKeyword` derived from the test scenario (e.g., `auth`, `cash-ins`). A new sorting algorithm (`sortFilesByTargetedRelevance`) will prioritize:
1. Routing Constants (`routes.ts`, `endpoints.ts`)
2. Exact-match API files (`src/api/{keyword}/api.ts`)
3. Exact-match UI files (`src/pages/{keyword}/*.tsx`)
4. All other files. 
This guarantees the LLM receives the exact API definitions and route constants needed within the 30-file / 80k-token limit.

## [x] 3. Payload Test Data
*Status: Resolved*

**Solution:** Implement **Dynamic API Chaining & UI Context Extraction**.
Because QA engineers test the "Full Flow" (creating prerequisite data from scratch), the LLM will generate a sequential chain of `api_request` steps.
* **Variable Extraction:** The prompt will instruct the LLM to use a generalized response extraction syntax: `{{STEP_X_RESPONSE.path.to.field}}`. For example, if Step 2 creates a supplier, Step 3 can use `{{STEP_2_RESPONSE.data.id}}` in its payload.
* **Dropdown / Lookup Emulation:** The LLM will leverage the injected UI component code (`.tsx`) to identify required lookup data. If a form requires a `tax_id` from a dropdown, the LLM will generate a `GET /taxes` request first, and use `{{STEP_X_RESPONSE.data[0].id}}` to populate the subsequent POST payload.
* **Mock Data Inference:** For raw string/number fields (e.g., amounts, names), the LLM will invent realistic mock data based on the injected TypeScript types, unless explicitly provided in the test case's `InputData`.

## [x] 4. Runner Handoff (Execution)
*Status: Resolved*

**Solution:** Define a strict contract between the Go backend (generator) and the Playwright/Cypress runner (executor).

1. **Dynamic Base URL:** The API Base URL will not be hardcoded by the LLM. Instead, the frontend UI for Test Generation will include an input field for the user to specify the `API Base URL` for that specific project/environment. This value is passed to the Go backend and forwarded to the test runner. The runner will prepend this Base URL to the `apiEndpoint` path generated by the LLM.
2. **Framework Metadata for Browser Injection:** The Go backend will analyze the fetched codebase (looking for `vite.config.ts` vs `next.config.js`) and append a `"framework": "vite" | "nextjs"` field to the top-level `GeneratedAutomation` JSON object. 
3. **Execution Handoff Logic:** The runner engine must:
   * Instantiate a memory dictionary (`stepResponses`).
   * Perform regex variable hydration (`{{AUTH_TOKEN}}`, `{{STEP_X_RESPONSE...}}`) before firing any HTTP request.
   * Execute the `api_request` and store the JSON response in memory.
   * Use the `"framework"` metadata to determine *how* to inject the auth state (Cookies vs `storageState.json`) before executing the first UI action (`navigate`).
